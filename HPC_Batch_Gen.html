HPC Batch Script Builder

Generate a batch script that uploads, runs, and retrieves HPC jobs from Windows to Linux via PuTTY (pscp / plink).

1Connection & PuTTY paths
2Copy files  Windows → Linux
Note — For browser-security reasons the full path can't be read, so only the file name is captured. The generated script assumes these files sit in the same folder as the .bat (%~dp0). Add wildcards like *.py or *.inp via manual entry.
3Linux working directory
Use this when running multiple jobs in parallel: giving each run its own numbered subfolder keeps parallel runs from reading and writing the same files at once — each run is isolated by its run number.
4Commands to run on Linux  (example)
One per line, chained in order with && (stops if a command fails). The example below runs the pre-processing script, submits the job to the queue in the foreground and waits for it to finish, then runs the post-processing script to generate the output.
Job-submission syntax depends on your cluster's scheduler (LSF, PBS) or a site wrapper script, so the exact commands will differ — refer to your internal / company documentation for the correct submission commands.
5Copy files  Linux → Windows
Remote file to pull back and the local folder to save it in. Leave the local folder empty to save next to the .bat (%~dp0).
Advanced options
Why tmux? A tmux session keeps the job alive on the HPC even if your PuTTY / SSH connection drops. The script launches it detached and then polls every 30 s until it finishes, so a network hiccup on the Windows side won't kill the run.

Foreground assumption. This script assumes the job runs in the foreground (e.g. Abaqus's interactive option, or a :wait queue) so the session stays occupied until the job completes and the wait logic can detect it. If the job is sent to the background (the submit command returns immediately while the solver keeps running under the scheduler), the session won't hold onto it and the wait / cleanup steps won't work — develop a separate script for that case.
